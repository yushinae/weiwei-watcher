# 监控页面重新设计与重构计划

## Summary（目标概述）
对现有监控页面（`/monitor`）进行重新设计与重构，在不改变核心业务含义的前提下，提升：
- **信息架构**：全局过滤/时间范围/到期维度统一入口，减少分散的“各图各控”状态。
- **视觉与一致性**：严格对齐项目 Design System（暗色交易终端），清理 raw hex，建立监控页局部识别点（“Signal Accent”）。
- **交互与可用性**：更清晰的 drill-down（点击图表元素→侧栏详情/跳转），更好的空态/错误态/加载态。
- **性能**：大表格虚拟化、渐进渲染、减少无关重渲染，为后续接入实时数据打基础。

> 设计方法遵循 `frontend-design`：明确美学方向、字体与排版有主张、主题用 CSS 变量/Token 统一、动效克制但精致、避免模板化“后台 UI”观感。

---

## Current State Analysis（现状分析）

### 1) 技术栈与路由现状（以代码库为准）
- 项目为 **Vite + React 19 + React Router**（`package.json` 的 `dev/build` 为 `vite`；`src/main.tsx` 使用 `BrowserRouter`）。
- 监控页路由入口：`/monitor` → `src/pages/MonitorPage.tsx`（在 `src/App.tsx` 的 `<Routes>` 中定义）。

### 2) 监控页实现与重复资产
- `src/pages/MonitorPage.tsx`：
  - 单文件包含：mock 数据、SVG 图表实现、表格实现、Tab 框架、多个 coin state（`smileCoin/skewCoin/...`）。
  - 视觉大量使用 raw hex（如 `bg-[#0A0A0D] border-[#1A1A22]`），与设计系统要求（语义 token 优先）冲突。
- `src/registry/monitorWidgets.tsx`：
  - 存在大量与 MonitorPage **重复**的 mock 数据与图表实现（Vol/Smile/Skew/VRP/IVRank/Surface/Dist/Polymarket 等）。
  - 更像“可复用 Widget 组件库”，但与页面实现分叉，后续维护会“双改漂移”。

### 3) 相关依赖与可复用能力
- 图表：`recharts`（多处使用）；`lightweight-charts`（已有封装 `src/components/LightweightChart.tsx`）。
- 虚拟化：`@tanstack/react-virtual` 已在 `src/pages/OptionsChainPage.tsx` 使用（可复用模式）。
- 网格工作台：`react-grid-layout` 已用于 `src/pages/DashboardPage.tsx`（可复用“终端式”布局密度与交互范式）。
- 动效：`motion` 已用于 App Shell（可用于抽屉/切换动效的精细化）。
- 设计系统：`docs/design-system.md` 指定 token、排版（`.tnum/.data-num`）、滚动与 WidgetCard 规范。

### 4) 主要问题（从“重新设计”视角）
1. **信息架构割裂**：多个 Tab 覆盖范围广，但缺少统一的全局过滤与维度联动（coin/range/tenor 等），用户需要反复在不同区域“找控制”。
2. **状态分散**：每个 Panel 各自维护 coin state，缺少一致的“当前分析上下文”。
3. **视觉 token 不统一**：raw hex 散落，违背“语义 token 优先”，导致 UI 漂移风险高。
4. **性能与演进风险**：大表格无虚拟化/无骨架屏；后续接实时数据容易出现频繁重渲染与布局跳变。

## Proposed Changes（改造方案与文件级变更）

### A. 设计与交互方向（“Quiet Terminal + Signal Accent”）
**基调**：维持“暗色交易终端”的克制密度（背景层级 + 细边线 + 小字号可读性），并为监控页增加两个可记忆细节：
1) **Signal Accent（信号高亮）**：全局控制条/选中态不使用纯色细线，改为“细长渐变滑块 + 轻微 glow（品牌蓝）”，并在 hover/active 上用统一 easing。
2) **双资产语义点缀**：BTC 暖色（amber 系）、ETH 冷色（blue 系）仅用于 chip/badge/小强调，避免大面积色块。

**落地规则**：
- 统一采用 Design System 的 token class（`bg-bg-card / border-border-subtle / text-brand-blue / text-trade-up/down`），监控页只允许极少量局部变量（作用域容器 `.monitor-scope`）。
- 数字强制 `.tnum`；关键数字可使用 `font-mono`（若已有等宽字体）但“点到为止”。
- 动效统一：短时长（120–180ms）、同一曲线（与现有终端一致），避免夸张位移。

### B. 信息架构：从“Tab + 各自状态”到“统一上下文 + 结构化内容”

#### B1) 新增全局控制条（Sticky Header）
新增 `MonitorHeader`，提供统一入口：
- 资产（coin）：BTC/ETH
- 时间范围（range）：`24H / 7D / 30D / 90D / 自定义（先 UI 占位）`
- 到期/tenor（tenor）：`7D/14D/30D/60D/90D`（影响 smile/surface/skew 等）
- 视图（view）：保留“Tabs”信息架构，但让每个 Tab 内容遵循一致布局（卡片化/网格化）
- 操作：刷新（先前端刷新 mock state）、重置布局（可选）

#### B2) 统一分析上下文与 URL 同步
新增 `useMonitorQueryState()`：
- 将 `coin/range/tenor/tab` 同步到 URLSearchParams（可复制链接、刷新不丢）。
- 将当前 drilldown 选择（selection）保存在内存 state（也可后续选择性写入 URL）。

#### B3) Drilldown（Inspector 抽屉）
新增 `InspectorDrawer`：
- 点击 Smile 点 / Surface 单元 / Skew 行：设置 selection，并打开右侧抽屉展示详情（含维度、局部趋势、关联数值）。
- 提供“跳转到期权链”按钮（路由跳转到既有 `OptionsChainPage`，并携带 query 参数，如 coin/tenor）。

### C. 架构重构：消除重复实现，页面变薄、组件可复用

#### C1) 引入 feature 目录（只做监控域内收敛）
新增目录：`src/features/monitor/`（不影响其他页面结构）
- `components/`
  - `MonitorHeader.tsx`
  - `MonitorLayout.tsx`（Tab 内容的统一布局骨架：Grid + WidgetCard 容器）
  - `InspectorDrawer.tsx`
  - `Skeletons.tsx`（终端质感 skeleton：继承卡片层级、使用项目已有 shimmer）
  - `VirtualizedTable.tsx`（封装 `@tanstack/react-virtual` 行虚拟化 + sticky header）
  - `widgets/`（从 MonitorPage / monitorWidgets 中拆出的可复用图表/表格卡片组件）
- `hooks/`
  - `useMonitorQueryState.ts`
  - `useMonitorSelection.ts`
  - `useMonitorData.ts`（统一数据入口：先接 mock，后续可切换实时/快照）
- `data/`
  - `mock.ts`（集中所有 mock 数据；MonitorPage 与 registry 都从这里取）
- `types.ts`（Coin/Range/Tenor/Selection 等类型）

#### C2) MonitorPage 变为装配层（组装 + 路由状态）
**目标文件**：`src/pages/MonitorPage.tsx`
- 移除：大段 mock 数据、SVG helper、表格/图表实现、各 Panel 各自 coin state。
- 保留：路由入口组件本身，负责：
  - 调用 `useMonitorQueryState()` 拿到 `tab/coin/range/tenor`
  - 渲染 `MonitorHeader`
  - 渲染 `MonitorLayout`（按 tab 切换对应 widgets）
  - 挂载 `InspectorDrawer`
- 滚动策略：对齐设计系统建议，优先使用 `ElasticLayout` 作为页面滚动容器，避免把 `overflow-y-auto` 直接挂在页面根上造成体验不一致。

#### C3) 统一 Widgets 来源，避免“双份实现”
**目标文件**：
- `src/registry/monitorWidgets.tsx`
- `src/registry/index.tsx`

改造方式：
- 将现有 `MonitorPage.tsx` 与 `monitorWidgets.tsx` 中重复的图表/表格/卡片组件拆到 `src/features/monitor/components/widgets/*`，并统一从 `src/features/monitor/data/mock.ts` 取数据。
- `monitorWidgets.tsx` 退化为“从 feature re-export + 轻装配（如传默认尺寸/标题）”，`registry/index.tsx` 的注册逻辑保持不变，只调整 import 来源。

---

### D. 视觉落地：Token 化 + 局部变量作用域

#### D1) 监控页局部变量（不污染全局）
**目标文件**：`src/index.css`
- 新增局部作用域（示例类名）：`.monitor-scope { ... }`
- 建议新增的最小变量集（示例，具体值以设计系统 token 组合为准）：
  - `--monitor-signal-glow`（品牌蓝 glow）
  - `--monitor-accent-btc` / `--monitor-accent-eth`（小面积点缀色）
  - `--monitor-gridline`（弱分隔线，用于图表网格/表格分隔）

#### D2) 清理 raw hex，统一语义 token
**目标文件**：
- `src/pages/MonitorPage.tsx`（重构后大幅变薄）
- `src/features/monitor/**/*`
- `src/registry/monitorWidgets.tsx`（必要的 token 化同步）

规则：
- 新增/重构代码中不再引入 `bg-[#...] border-[#...] text-[#...]`；优先使用 `docs/design-system.md` 中的 token class。
- 对图表 stroke/fill 等“必须是颜色值”的场景：优先用 CSS 变量（例如 `stroke="var(--color-brand-blue)"` 或局部 `--monitor-*`），避免散落常量。

---

### E. 性能：虚拟化、渐进渲染、减少重渲染

#### E1) 虚拟化策略
**目标文件**：`src/features/monitor/components/VirtualizedTable.tsx`
- 以 `@tanstack/react-virtual` 为基础封装“行虚拟化 + sticky header + 列对齐”。
- 参考 `src/pages/OptionsChainPage.tsx` 的 `useVirtualizer` 使用方式，保持工程风格一致。

#### E2) 渐进渲染与骨架屏
**目标文件**：`src/features/monitor/components/Skeletons.tsx`
- 首屏优先渲染布局骨架（Header + WidgetCard 容器），数据 ready 后用短动效（`opacity/blur`）填充。
- Skeleton 视觉：不要“大块灰条”，而是继承卡片边线层级，并复用项目已有 shimmer（`src/index.css` 已存在相关样式）。

#### E3) 减少无关重渲染
**目标文件**：`src/features/monitor/components/widgets/*`
- widgets 拆分为纯展示组件，并通过 `React.memo`、稳定的 props（避免 inline object/function）减少 tab/coin 切换时的重渲染面积。
- 若后续接入 zustand 实时数据：使用 selector + shallow，只订阅必要字段。

## Assumptions & Decisions（关键假设与决策）
1) 用户描述为 “Next.js/React”，但仓库实际为 **Vite + React Router**；本计划以当前代码库为准执行，**不做框架迁移**。
2) 监控页当前数据均为 mock；本计划**不引入真实数据接口**，但会通过 `useMonitorData()` 固定数据入口，方便未来替换为 ws/fetch。
3) 信息架构选择：**保留 Tabs 作为一级导航**（降低一次性认知迁移风险），但统一为“全局控制条 + 卡片化网格布局 + Inspector drilldown”的页面结构。
4) 仅做“监控域内”局部变量（`.monitor-scope`），不新增/重写全局设计系统（避免 token 体系发散）。

## Verification（验收与验证步骤）

### 1) 功能与交互
1. 打开 `/monitor`：首屏先出现 Header 与卡片 skeleton，随后内容渐进填充，无明显跳变。
2. 切换 coin（BTC/ETH）：
   - URL query 同步更新；刷新页面后状态保持；
   - 所有与 coin 相关的 widgets 同步切换。
3. 切换 range/tenor：
   - 影响对应图表与表格（mock 维度可先做“选择态 + 标注”，数据可暂不变但结构必须可扩展）。
4. Drilldown：
   - 点击 Smile/Surface/Skew 的元素 → 打开 InspectorDrawer，展示 selection 信息；
   - 点击“跳转到期权链”可导航至期权链页面，并携带 coin/tenor query（若实现）。

### 2) 视觉一致性（对齐 Design System）
1. 监控页新增代码中不应出现大量 raw hex；优先使用 token class（参考 `docs/design-system.md`）。
2. 数字列对齐：关键数值/表格必须使用 `.tnum` / `.data-num`。
3. 动效统一：hover/切换/抽屉的时长、曲线保持一致，且克制。

### 3) 性能与稳定性
1. 构造大数据（mock 扩展）：表格滚动流畅（虚拟化生效），不因大量行渲染卡顿。
2. React Profiler 检查：切换 coin/range 时，仅相关 widgets 发生重渲染，Header/无关模块不应频繁更新。
