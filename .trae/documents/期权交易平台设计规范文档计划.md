# 期权交易平台：设计规范文档（design-system.md）实现计划

## Summary（摘要）
在项目内新增一份面向协作开发者的 **UI 设计规范/风格指南**：`docs/design-system.md`。该文档将基于现有代码“提炼事实”，系统化描述网站的整体风格、页面/布局尺寸、响应式规则、组件规范、交易与图表 UI 规范、以及工程层面的写法约束（含 shadcn 的语义化与组合式原则），以便朋友可在不反复阅读源码的情况下继续扩展与保持一致性。

## Current State Analysis（现状分析）
### 技术与样式体系（从仓库读取到的事实）
- 项目形态：Vite + React + React Router（见 `package.json` / `src/main.tsx` / `src/App.tsx`）。
- 样式体系：Tailwind CSS v4（通过 `@tailwindcss/vite` 插件 + `src/index.css` 内 `@import "tailwindcss";` 与 `@theme { ... }` 配置 token）。
- 主题倾向：暗色交易终端风格，强调分层背景（base/dim/deep/card/surface）、低对比边线、交易涨跌色、数字对齐（tabular-nums）。

### shadcn Skill 关键约束（需融入到设计规范文档的“工程规则”部分）
> 说明：仓库未发现 `components.json`，因此此处引用 shadcn 的“设计系统/工程规范”思想，而非要求执行 shadcn CLI 初始化。
- 使用语义化颜色（`bg-primary` / `text-muted-foreground` 这种语义思路），避免到处散落 raw hex；在本项目中对应做法是：优先使用 `@theme` 中已存在的 token（如 `bg-bg-card`、`border-border-subtle`、`text-brand-blue`、`text-trade-up/down`）。
- layout 间距用 `gap-*`，避免 `space-x-* / space-y-*`。
- 用 `cn()` 管理条件 class，不用手写字符串三元拼接。
- Overlay（Dialog/Sheet/Popover 等）避免在业务组件到处手写 z-index；应有统一的层级约定（即使暂时仅写在文档里，后续再收敛代码）。
- 组合优先：优先复用现有组件/容器，而非手搓 div 样式（在本项目体现为：WidgetCard/Popup/ElasticLayout 等作为“基础构件”）。

## Proposed Changes（改动方案）
### 1) 新增设计规范文档：`docs/design-system.md`
**新增文件**
- `docs/design-system.md`

**目标**
- 把“现状 UI 风格”以可执行规范形式沉淀下来：不仅描述“长什么样”，还要描述“怎么写出来、如何验收一致性”。

**文档结构（必须落地到 md 的目录与小节中）**
1. 概述（Scope & Philosophy）
   - 暗色交易终端风格定位：分层背景、低对比边线、数值优先与交易语义色。
   - 说明“本文是事实提炼 + 约束性规范”，适用于后续新增页面/组件。
2. Design Tokens（全局 token + 局部域 token）
   - 逐项列出 `src/index.css @theme` 中 token：Brand、Surface、Borders、Trade colors、radius 等，并给出用途与示例。
   - 明确“全局 token”与“局部主题域 token”（例如 `.db-oc-root` 的 `--db-*` 仅限期权链域）。
   - 给出“语义别名映射表”（写在文档里）：如 background/card/border/primary/success/destructive 等语义如何对应当前 token（不要求立刻改代码，但为未来演进铺路）。
3. Typography（字体与数字规范）
   - 字体栈（`--font-sans` 等）与应用范围（UI 文本 vs 数据文本）。
   - tabular-nums 规则：`.tnum`、`.data-num`、`.strike-num` 的使用场景清单（价格、执行价、盈亏、希腊值、盘口数量等）。
4. Layout & Sizing（页面大小、布局与尺寸基线）
   - App Shell：`h-screen`、header 高度（48px）、main 的 overflow 策略、常用 padding/gap。
   - Dashboard 网格：断点/列数/rowHeight/margin/padding；Widget 默认尺寸与最小尺寸写法约定。
   - ElasticLayout：restGap（默认 6px）、上下分隔条色值、滚动检测、适用页面类型。
5. Components（组件规范：形态、状态、交互）
   - WidgetCard：视觉（渐变描边+阴影+hover 上浮）、结构（标题/操作/内容 padding）、fullscreen 变体。
   - Button/IconButton/Pill：高度基线（例如 26/28/32/36/40/48 等从代码里归纳）、圆角基线（6/8/10/12）、hover/active 动效曲线与时长（现状大量 `120ms` + `cubic-bezier(0.22,1,0.36,1)`）。
   - Overlay（Popover/Modal/Drawer）：统一层级表（写入文档），键盘 Esc、click-away、backdrop blur 等规则；强调“业务侧避免新增 z-index 孤岛”。
   - Ant Design（仅在 TradeLog 场景）：主题注入与 CSS 覆写边界（哪些走 token，哪些必须 CSS；把现状 `.trade-log-table` 规则写成可复用模板）。
6. Charts & Trading UI（交易与图表规范）
   - Recharts：grid/tooltip/渐变填充的规范化配色；统一 tooltip 的容器样式与字体（尽量映射到 token）。
   - lightweight-charts：通过 `colors` 注入的标准接口（在文档里定义 `ChartColors` 约定，并映射到全局 token）。
   - 订单簿/盘口/行情：涨跌闪烁与强调色规则；禁止硬编码颜色（优先 `trade-up/down`）。
   - 期权链（Deribit 风格）：`.db-oc-root` token 表、行高/字号/padding、hover/selected、spot 紫色与提示规则；把 `reference/deribit-desktop/pixel-spec.md` 作为参考/附录链接。
7. Engineering Rules（工程写法约束：把规范变成“可执行的 code review checklist”）
   - className：必须 `cn()`；条件分支不要手写复杂模板字符串。
   - spacing：新增代码禁用 `space-x/y-*`，统一 `gap-*`。
   - token：新增代码禁止随意写 raw hex；若必须（临时实验），必须在旁边标注原因与 TODO。
   - overlay：新增代码禁止手写 `z-[...]`（或至少必须引用“层级表”里的统一常量/约定）。
   - a11y：输入控件 label/aria、focus-visible、键盘可达（以 checklist 形式写入）。
