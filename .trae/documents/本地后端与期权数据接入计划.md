# 本地后端与期权数据接入计划（Deribit + Bybit）

## Summary（目标概述）
为当前 Vite + React 项目新增**本地可部署**的后端与数据库能力，重点完成 **Deribit + Bybit 期权数据**的接入、聚合与落库，并提供前端可用的 REST + 推送通道（SSE/WS），用于：
- 期权链快照（按标的/到期）
- IV 面 / 偏斜指标（surface + skew metrics）
- 成交流（聚合为主）
- 订单簿（内存合并 + 采样聚合落库）
- 采集连接状态（终端风 UI 入口）

技术决策（来自澄清结果）：
- 后端：**Node.js + Express**
- 数据库：**PostgreSQL**
- 更新方式：**WebSocket 实时 + 定时快照落库**

## Current State Analysis（现状分析）

### 1) 仓库现状：后端代码缺失但依赖已存在
- `package.json` 中已包含 `express` 依赖，但代码库未发现 `express()` / `app.listen()` 等后端入口，当前为纯前端运行模式。
- 现有 WebSocket 逻辑位于前端 `src/App.tsx`：连接 Binance miniTicker，包含断线重连与批量更新策略（可作为“前端订阅后端状态流”的参考模式）。
- `vite.config.ts` 目前未配置 dev proxy；引入本地 API 后需要补充。

### 2) 关键约束：期权数据高频与数据量
你希望存：**期权链快照 + IV 面/偏斜 + 成交流 + 订单簿深度**。其中订单簿与 ticker 频率高，必须采用：
- 内存合并（snapshot + delta）+ 采样聚合（1s/500ms）落库
- 避免逐条 delta 全量落库（会迅速撑爆写入与空间）

### 3) 外部接口要点（来自调研）
- Bybit WS v5 public option：连接/心跳/订阅格式与 orderbook snapshot/delta 合并规则明确；ticker 在 Option 场景下推送频率可达 100ms。
  - https://bybit-exchange.github.io/docs/v5/ws/connect
  - https://bybit-exchange.github.io/docs/v5/websocket/public/orderbook
  - https://bybit-exchange.github.io/docs/v5/websocket/public/ticker
  - https://bybit-exchange.github.io/docs/v5/websocket/public/trade
- Deribit Market Data Best Practices：强调窄订阅、避免 REST 轮询、使用 instrument.state 驱动订阅维护、丢序检测与必要的 resync。
  - https://support.deribit.com/hc/en-us/articles/29592500256669-Market-Data-Collection-Best-Practices

#### A2. 后端依赖（决定稿）
在 `server/package.json`（或根 package.json）中加入：

- 工具：`dotenv`（仓库已存在，可复用），`tsx`（仓库已有，可复用）

### B) PostgreSQL Schema（落库模型 + 索引/分区）
目标：兼顾**前端快速读取**与**后续分析/回放**，同时控制高频数据写入成本。

#### B1. 迁移文件（新增）
`server/src/db/migrations/*.sql`（或 node-pg-migrate 生成的 ts/js）

#### B2. 表结构（决定稿）
1) `instruments`（维表）
- `exchange`, `symbol`（unique）
- `base`, `quote`, `expiry_ts`, `strike`, `option_type`
- `status`, `created_at`, `updated_at`
- 索引：`unique(exchange, symbol)`, `btree(exchange, base, expiry_ts)`, `btree(exchange, base, option_type, expiry_ts, strike)`

2) `option_chain_snapshots`（按 ts 分区；前端主读）
- `ts, exchange, base, expiry_ts, underlying_price/index_price, payload jsonb`
- 主键：`(exchange, base, expiry_ts, ts)`
- 索引：`btree(exchange, base, expiry_ts, ts desc)`, `brin(ts)`

3) `iv_surface_points`（按 ts 分区）
- `ts, exchange, base, expiry_ts, moneyness, delta, iv`
- 索引：`btree(exchange, base, expiry_ts, ts desc)`, `btree(exchange, base, ts desc)`, `brin(ts)`

4) `iv_skew_metrics`（按 ts 分区）
- `ts, exchange, base, expiry_ts, atm_iv, rr25, fly25, ...`
- 索引同上

5) `option_trades_1s`（按 bucket_ts 分区，聚合表）
- `bucket_ts, exchange, symbol, count, buy_qty, sell_qty, vwap, min_price, max_price`
- 主键：`(exchange, symbol, bucket_ts)`
- 索引：`btree(exchange, symbol, bucket_ts desc)`, `brin(bucket_ts)`

6) `orderbook_samples_1s`（按 ts 分区，采样聚合）
- `ts, exchange, symbol, best_bid, best_ask, mid, spread, bid_depth_n, ask_depth_n, imbalance_n, levels jsonb?`
- 主键：`(exchange, symbol, ts)`
- 索引：`btree(exchange, symbol, ts desc)`, `brin(ts)`

7) `collector_status`（小表，连接状态）
- `source primary key`, `state`, `last_msg_ts`, `msg_rate_1m`, `last_error`, `updated_at`

### C) 采集器与聚合器：WS 实时 + 采样/快照落库
总体原则：**Collector 只负责“连上+解析+发事件”**；聚合与落库在 `agg/` 与 `jobs/`。

#### C1. 统一内部事件类型（新增）
`server/src/collectors/types.ts`
- `InstrumentDiscovered`
- `TickerUpdate`
- `Trade`
- `OrderBookSnapshot`
- `OrderBookDelta`

#### C2. Bybit 接入（新增/实现）
文件：
- `server/src/exchange/bybit/ws.ts`
- `server/src/exchange/bybit/http.ts`（用于启动时 instrument bootstrap、断档补快照）
- `server/src/exchange/bybit/normalize.ts`

关键实现点（决定稿）：
- WS endpoint：`wss://stream.bybit.com/v5/public/option`
- 按官方规则合并 orderbook delta：size=0 删除；不存在则插入；存在则更新（并维护 topN）。
- topic 建议：
  - `tickers.{symbol}`（高频 100ms；用于链快照/IV/Greeks）
  - `orderbook.25.{symbol}`（高频 delta；只内存合并）
  - `publicTrade.{baseCoin}`（交易流；进聚合器）
- 自动重连 + resubscribe；心跳 20s ping 保活。

#### C3. Deribit 接入（新增/实现）
文件：
- `server/src/exchange/deribit/ws.ts`（JSON-RPC 2.0 subscribe）
- `server/src/exchange/deribit/http.ts`
- `server/src/exchange/deribit/normalize.ts`

关键实现点（决定稿）：
- 启动时 REST bootstrap instruments；
- 订阅 `instrument.state.option.{currency}` 驱动新增/过期 instrument 的订阅维护；
- 订阅采用“窄订阅策略”，并实现丢序检测与必要的 resync（按 best practices）。

#### C4. 内存态与采样
文件（新增/实现）：
- `server/src/agg/orderbookStore.ts`：按 symbol 维护 orderbook（snapshot+delta 合并）
- `server/src/agg/tradeAggregator.ts`：将 trades 聚合到 1s bucket
- `server/src/agg/sampler.ts`：每 1s 从 orderbookStore 采样并生成 `orderbook_samples_1s` 记录
- `server/src/agg/chainBuilder.ts`：由 per-instrument tickers 构建 expiry 维度 chain payload
- `server/src/agg/ivSurface.ts`：从 chain 计算/提取 surface points 与 skew metrics

#### C5. 定时任务（快照落库）
文件（新增/实现）：
- `server/src/jobs/scheduler.ts`：setInterval + `pg_advisory_lock` 防止多实例重复跑
- `server/src/jobs/snapshots.ts`：
  - `chain snapshot`（5s/10s）：写 `option_chain_snapshots`
  - `iv surface + skew`（10s/30s）：写 `iv_surface_points`/`iv_skew_metrics`
  - `trades 1s`：写 `option_trades_1s`
  - `orderbook 1s samples`：写 `orderbook_samples_1s`

## Assumptions & Decisions（关键假设与决策）
1) 初期本地部署不做鉴权（仅限 localhost）；但仍会加最小安全基线：CORS 限制、rate limit、WS/SSE 连接数限制。
2) 订单簿采用**内存合并 + 采样聚合落库**，不做逐 delta 落库（成本过高）。
3) 成交流默认落 **1s 聚合**；如后续确需 raw trades，再增加 raw 表与写入开关。
4) IV 面/偏斜可先由快照计算（从链 payload 提取点/插值），不依赖交易所“直接给 surface”。
5) 本计划默认覆盖标的 BTC/ETH；如你需要更多标的，在 `server/src/config.ts` 增加可配置列表即可扩展。
